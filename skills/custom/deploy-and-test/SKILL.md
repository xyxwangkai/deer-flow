# 部署与测试闭环 Skill

> 扩展说明：`heartbeat` 推荐请求模板、测试会话常量初始化步骤、IDL 版本选择方法、双集群排障路径、以及 `log_id` 关联日志观测 SOP，统一沉淀在同目录 `EXTEND.md`。当任务重点是 PPE 泳道验证、低风险 heartbeat 冒烟、或需要快速搭建 RPC + 日志闭环时，优先同时参考 `EXTEND.md` 与 `templates/` 下的命令模板。

## ⚠️ 第零步：部署前置门禁（必须首先执行）

**在做任何测试之前，必须通过以下决策树判断是否需要部署：**

```
本次对话中是否修改过代码（Edit/Write/创建新文件）？
  ├─ 是 → 【必须部署】无论之前是否部署过，本地代码已变更，远端是旧代码
  └─ 否 → 本次对话中是否已成功部署过？
              ├─ 是 → 可以直接测试
              └─ 否 → 用户明确说不需要部署？
                        ├─ 是 → 直接测试
                        └─ 否 → 【必须部署】
```

**绝对禁止**：改过代码后跳过部署直接测试。这会导致在旧代码上做无效测试，浪费所有测试调用。

## 适用场景
- Claude 修改代码后，需要验证代码在真实 PPE 泳道环境中的正确性
- 需要自动化 “修改 → 部署 → 测试 → 日志观测 → 验证” 的完整闭环
- 接口行为变更后需要端到端验证
- 需要基于 `log_id` 追踪服务端处理链路，判断问题是否由本次修改引入
- 需要确认目标 PPE 服务实际运行版本、镜像 tag 或启动构建信息

## 完整流程概览

```
第零步：部署门禁检查（改过代码？→ 必须部署）
    ↓
第一步：确认验证目标（PSM / env / lane / 期望版本）
    ↓
第二步：查询实例分布（按 cluster / zone / idc 拆开）
    ↓
第三步：锁定正确 IDL 版本（不要凭经验猜）
    ↓
第四步：优先对 Running 实例打 heartbeat 冒烟
    ↓
第五步：提取 log_id，进入 Argos / 实例日志观测
    ↓
第六步：必要时对多个 cluster 分别验证并对比结果
    ↓
第七步：结合日志 / 镜像 / 构建信息确认版本
    ↓
输出测试报告与版本结论
```

## 第一步：部署到 PPE

### 1.1 前置条件
- 已具备可用的 `bytedcli` 部署入口
- 已通过飞书扫码登录（token 有效期内无需重复登录）
- 已明确目标 PSM、PPE 泳道名，以及部署链路所需的项目参数
- 已明确日志查询入口或日志平台检索方式，用于后续 `log_id` 观测

### 1.2 工具选择原则
部署与环境操作统一以 **bytedcli** 为唯一入口：
- 部署使用 `bytedcli`
- 环境 / 泳道定位使用 `bytedcli`
- 实例信息查询使用项目标准控制面能力或 `bytedcli`
- RPC 验证使用 `bytedcli api-test rpc-call`
- 日志观测使用项目日志平台、聚合日志检索能力或 bytedcli 返回的日志直达链接

**不再使用其他已下线或非标准部署工具。**

### 1.3 本地编译说明
**本地 `go build` 可能因工具链版本不匹配而失败**（如 `compile: version "go1.25.4" does not match go tool version "go1.24.9"`）。这是本地环境问题，不代表代码有错。

处理策略：
- **不要**花时间排查本地工具链问题
- **以远端标准构建结果为准**——优先参考 bytedcli 对应部署链路的远端编译结果
- 如果远端编译成功，说明代码没有语法 / 编译问题
- 如果远端编译失败，提取错误信息修复代码

### 1.4 定位部署目标
如果尚未确定目标 PPE 泳道的部署目标、实例或运行单元信息，先通过项目标准入口或 `bytedcli` 查询目标环境状态，确认：

- PSM
- PPE env / lane
- cluster
- zone
- idc
- 当前运行单元 / 实例标识（若日志平台检索需要）
- 实例是否为 `Running` 而非仅 `NotReady`

### 1.5 部署命令
**统一使用 bytedcli 对应项目标准部署命令。**

示意：

```bash
bytedcli <project-specific deploy command>
```

要求：
- 命令必须明确指向目标 PPE 泳道
- 命令输出中应能看出构建、发布、收敛的关键阶段
- 若项目提供 deploy / upgrade / rollout status 等子命令，优先使用标准链路，不自行拼装旁路命令

### 1.6 部署执行策略
**必须使用子 Agent（Task tool）在后台执行部署**，因为部署耗时较长（首次约 10 分钟，后续约 3 分钟）。

部署等待期间，主线程应并行做三件事：
1. 用 `db-query` 准备测试数据
2. 初始化测试会话常量
3. 查询实例分布并预判应优先验证哪个 cluster

### 1.7 部署失败处理
**编译失败**：提取错误信息，分析是否是本次代码修改引入的问题，修复后重新部署。  
**服务启动失败**：查看部署日志与运行日志，分析 panic、配置错误、依赖异常。  
**登录过期**：出现飞书扫码二维码提示时，需用户手动扫码。

## 第二步：建立测试会话常量

每次测试会话开始时，**一次性获取并记录**以下公共参数：

- `psm`
- `func_name`（默认优先 `heartbeat`）
- `env`
- `cluster`
- `zone`
- `idc`
- `online`
- `idl_source`
- `idl_version`
- `runtime_unit`
- `log_entry`

**后续每个 test case 只变更 `req_body`，其他参数全部复用。**

## 第三步：先查实例，再决定打哪个 cluster

在 PPE 场景下，**不要默认 `cluster=default` 一定可用**。同一 PSM 可能同时分布在多个 cluster，且状态不同。

建议流程：
1. 先查实例列表
2. 按 cluster 聚合查看 `Running / NotReady` 状态
3. 优先选择 `Running` 的 cluster 做第一枪 heartbeat
4. 如果一个 cluster 不通，不要立刻判断服务不可用；要继续验证其他 Running cluster
5. 如果后续用户反馈某个 cluster 已修复，再单独复测该 cluster

**关键经验**：
- `61003` 往往先看路由目标是否选错 cluster
- `Running` 与 `NotReady` 的实例混在一起时，必须先分清楚再打
- 多 cluster 共存时，要保留每个 cluster 的 pod / log_id / 返回结果，便于横向对比

## 第四步：锁定正确的 IDL 版本

**IDL 版本必须显式确认，不要凭旧经验写死。**

建议顺序：
1. 先查可用 API / schema 版本
2. 优先选择与当前服务实现匹配的具体版本号
3. 如果出现“required field missing”“reader error”“返回结构无法解析”等错误，先怀疑 **IDL 版本不匹配**，不要立即判定服务异常
4. 调整 `idl_version` 后重新验证同一请求

**关键经验**：
- `idl_version=1.0.0` 不一定适配当前服务返回结构
- 如果切到 `1.0.18` 后请求成功，应把“IDL 版本不匹配”记为流程问题，而不是服务逻辑问题
- 一个成功的 heartbeat 样本，比多个 schema 不匹配的失败样本更有价值

## 第五步：默认探测接口为 heartbeat

如果目标是先确认部署是否生效、接口是否可达、版本行为是否切到目标泳道，**优先使用 `heartbeat`**。

原因：
- 请求体最小
- 业务副作用最低
- 更适合做部署后冒烟验证
- 更容易结合 `log_id` 做端到端链路追踪
- 更适合做 cluster / IDL / 路由排障

请求构造原则：
- 顶层传 `env`
- `Base.TrafficEnv.Env` 也传同一个 PPE 环境
- `message_id` 每次唯一
- 使用 `body-file` 保持请求体稳定可复用

## 第六步：按返回错误分类排障

### 6.1 常见错误优先判断
- `61003` / 找不到服务地址：优先排查 cluster / env / idc / zone 是否选错
- `1701 upstream request failed`：说明已命中实例，但服务内部、mesh ingress 或上游链路失败；需要进日志
- `required field missing` / `reader error`：优先排查 IDL 版本与服务返回结构不匹配
- `biz_status_code = 0`：说明 RPC 与 schema 基本打通，可进入日志 / 版本确认阶段

### 6.2 不同 cluster 返回不同错误时
如果不同 cluster 返回不同错误，不要简单合并结论，应分别记录：
- cluster 名
- pod 名
- request_address
- log_id
- biz_status_code
- 错误摘要

这通常意味着：
- 不同 cluster 的实例状态不同
- 或运行版本、发布进度、IDL 兼容情况不同

## 第七步：日志观测与版本确认

调用接口后，从响应中提取 `log_id`，进入日志平台检索。重点关注：
- 请求是否进入应用 handler
- 是否出现 `ERROR` / `WARN` / `panic`
- 启动日志中的版本号、镜像 tag、构建信息、git commit
- TCE / 部署面里 pod 对应的镜像版本

当目标是确认运行版本是否为某个版本（如 `1.0.0.909`）时，**最终证据优先级**建议如下：
1. pod 启动日志中明确打印版本号
2. TCE / 部署面显示的镜像 tag / 发布版本
3. 构建信息 / git commit 与发布记录可映射到目标版本
4. 业务行为与预期版本一致（只能作为辅助证据）

## 第八步：生成结构化测试报告

测试完成后，报告中至少应包含：
- PSM / env / cluster / zone / idc
- 实际命中的 pod
- 使用的 `idl_version`
- 请求样本与 `log_id`
- 每个 cluster 的结果对比
- 日志观测结论
- 是否已能确认目标版本
- 若未确认，还缺什么证据（版本日志 / 镜像 tag / 发布记录）

## 注意事项
1. **部署门禁是第一优先级**：改过代码必须先部署，绝不跳过
2. **先查实例状态，再打 heartbeat**：不要默认 `default` cluster 永远可用
3. **优先选择 Running 实例所在 cluster**：NotReady 实例不能作为有效验证目标
4. **idl_version 必须用具体版本号**：并在异常时优先怀疑 schema 不匹配
5. **env 必须顶层和 `Base.TrafficEnv.Env` 双写一致**：确保命中目标 PPE 泳道
6. **日志观测必须关联 `log_id`**：没有链路 ID 的测试结论不完整
7. **验证版本时，不要只停留在 RPC 成功**：要继续落到日志 / 镜像 / 构建信息
8. **多 cluster 结果要并列记录**：便于判断是路由问题、实例问题还是版本差异
9. **测试完成必须生成报告**：按结构化模板输出结论
