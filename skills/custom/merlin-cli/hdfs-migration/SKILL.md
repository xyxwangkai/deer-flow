# HDFS 数据传输

用于处理 HDFS 到 HDFS 的跨洋传输、跨地域同步和数据搬迁任务。这个 skill 不直接把四种操作混在一起讲，而是先识别用户意图，再读取对应 reference 文档执行。

## 适用范围

当用户的需求落在以下场景时触发本 skill：

- 在不同 HDFS 机房之间复制目录或文件
- 发起一条新的跨洋传输或数据同步任务
- 查看已有传输任务的状态、进度、失败原因或任务列表
- 查看单个传输任务下的传输子任务状态、目标路径、目标机房、失败原因
- 查询单个传输任务下传输子任务对应的 `Tiansuan` / `Zeus` 任务 ID
- 停止、取消、终止正在运行或排队中的传输任务
- 对失败、中断或需要重新执行的传输任务做重试

以下情况不要使用本 skill，应该先澄清或交给更合适的 skill：

- 本地文件和 HDFS 之间的上传下载
- 纯粹的数据表注册、DataCard、评估集导入
- 普通训练任务、评估任务或服务部署
- 用户只是泛泛问 “HDFS 是什么” 而不是要管理迁移任务

## 先判断操作类型

先根据用户意图分流，不要一上来就假设是“发起任务”。

| 操作类型 | 常见用户表达 | 读取文档 |
|------|------|------|
| 发起任务 | 发起、创建、提交、启动、拉起、同步一份数据、把 A 传到 B、多组路径批量同步 | `references/create.md` |
| 查看任务 | 查看、查询、列出、看状态、看进度、查失败原因、看任务详情、查看传输子任务、查目标路径、查 Tiansuan/Zeus 任务 ID | `references/query.md` |
| 停止任务 | 停止、取消、终止、kill、停掉这个同步 | `references/stop.md` |
| 重试任务 | 重试、重跑、重传、重新发起、失败后再来一次 | `references/retry.md` |

## 意图识别规则

优先用下面的信号判断用户想做什么：

1. 先看动作词：发起、查看、停止、重试
2. 再看用户是否提供了任务标识：`task_id`、`job_id`、任务链接、历史任务记录
3. 再看用户是否主要提供了资源信息：源路径、目标路径、目标机房

默认判断规则：

- 用户主要给的是源路径和目标路径，且没有现成任务 ID，优先判断为“发起任务”
- 用户给了任务 ID、任务链接或“这个任务”，优先判断为“查看 / 停止 / 重试”
- 用户说“失败了怎么处理”或“重新跑一下”，优先判断为“重试任务”
- 用户提到“看进度 / 看状态 / 为什么失败”，优先判断为“查看任务”
- 用户提到“传输子任务”“目标路径”“目标机房”“Tiansuan 任务 ID”“Zeus 任务 ID”，优先判断为“查看任务”，并在读取 `references/query.md` 后进一步下钻到传输子任务详情
- 如果一句话同时包含多个意图，先确认当前要执行哪一步，不要一次展开多个流程

## 缺参时怎么问

如果信息不足，先补齐本次操作的最小必要参数，只问最关键的问题。

先读取 `references/common-fields.md`，确认术语和公共字段的含义，再补问缺失信息。

建议遵循：

- 发起任务时，优先确认源路径、目标路径；必要时再根据目标路径推断或确认目标机房
- 发起任务如果有多组源 / 目标 路径对，优先让用户用 Markdown 表格或 JSON 列表一次性提供
- 发起任务在真正执行前，必须把最终的源 / 目标 路径对列表发给用户确认
- 查看任务时，优先确认是“单个 task_id”“多个 task_id”还是“按条件筛选”
- 查看任务如果没有 `task_id`，再让用户提供筛选条件：`creator`、任务创建时间范围、状态、HDFS 源路径
- 查看任务按条件筛选时，如果用户没有明确指定别人，默认将 `creator` 设为用户自己
- 查看任务按条件筛选时，分页默认 `limit=20`，最大不超过 `100`
- 查看传输子任务详情时，必须先有单个 `task_id`
- 停止 / 重试时，优先确认任务 ID；如果没有任务 ID，再让用户提供筛选条件
- 停止 / 重试前，必须先确认这些任务的 `creator` 是当前用户自己
- 不要替用户猜机房、路径或重试策略
- 如果涉及停止等高风险动作，执行前再次确认

## 推荐工作流

1. 判断这是 HDFS 跨洋传输任务管理需求
2. 识别操作类型：发起 / 查看 / 停止 / 重试
3. 读取对应 reference 文档
4. 补齐最小必要信息
5. 如果是查看任务，先判断是任务概览查询还是传输子任务详情下钻
6. 如果是发起任务且有多组路径对，先整理成表格或 JSON 列表，再给用户做最终确认
7. 先做一次参数校验或命令预览，再执行真实操作
8. 把结果按统一格式回复给用户

对应操作所需的 merlin-cli 工具、参数结构和结果展示方式，直接读取相应的 reference 文档即可，不需要再做工具发现。

## 输出格式

优先用下面的结构回复用户，避免只贴原始 JSON：

### 操作类型

发起任务 / 查看任务 / 停止任务 / 重试任务

### 已识别信息

- 已知的任务 ID、源路径、目标路径、机房、状态等

### 缺失信息

- 仍需用户补充的最小必要字段；如果没有缺失则写“无”

### 执行结果 / 下一步

- 发起成功时优先按 `references/create.md` 的约定，用 Markdown 表格展示任务信息，并在表格后补充目标机房、目标路径
- 查看成功时优先按 `references/query.md` 的约定，用 Markdown 表格展示主任务信息
- 如果用户明确要看传输子任务详情，在主任务表格之后追加第二张“传输子任务详情”表格
- 停止成功时优先返回本次实际停止的 `task_id` 列表，并用 Markdown 表格展示重新查询后的任务状态
- 重试成功时优先返回本次实际重试的 `task_id` 列表，并用 Markdown 表格展示重新查询后的任务状态
