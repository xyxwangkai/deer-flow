# 从吸收到技能化

本指南用于把 `eat` 产出的知识吸收结果，转化为真正可执行的 skill 资产。

## 一、判断是否需要新建 skill

满足以下 3 条中的至少 2 条，可考虑新建 skill：

- 有稳定触发词或触发场景
- 有明确的输入、工作流、输出结构
- 与现有 skill 相比，具有相对独立的目标边界

否则优先并入已有 skill。

## 二、五种常见沉淀形态

### 1. 写入 SKILL.md
适合：
- 新增触发词
- 固化步骤
- 约束输出结构
- 明确何时调用子资源

### 2. 写入 EXTEND.md
适合：
- 记录例外情况
- 保留评分细则
- 保存变体策略
- 补充长尾边界

### 3. 写入 references/
适合：
- 背景知识
- 术语表
- 对照表
- 方法详解
- 案例拆解

### 4. 写入 scripts/
适合：
- 可重复执行的解析/转换/导出动作
- 数据格式转换
- 从分析结论生成 JSON / Markdown / 脚手架

### 5. 写入 examples/
适合：
- 示例输入
- 理想输出
- 成功案例
- 对比案例

## 三、技能化的最小动作链

```text
知识源 -> 吸收结论 -> 原理/模式 -> 评估是否值得沉淀 -> 选择写入位置 -> 生成资产
```

## 四、推荐提问

在资料不足时，优先向用户追问：

- 你要的是一次性分析，还是长期沉淀？
- 你想并入现有 skill，还是独立建 skill？
- 你更关心方法论、脚本模板，还是触发式工作流？
- 这个知识后续会在什么场景重复使用？

## 五、质量门槛

只有同时满足以下条件，才建议进入长期技能库：

- 能说明解决的问题
- 能抽象出不止一个具体细节
- 能指出使用边界
- 能落到具体文件或脚本

## 六、输出模板

```markdown
## 吸收结论

## 关键原则

## 值得沉淀的资产

## 建议写入位置
- SKILL.md:
- EXTEND.md:
- references/:
- scripts/:
- examples/:

## 下一步
```
