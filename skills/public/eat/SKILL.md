# eat

把外部知识源吸收为 agent 的可复用能力，而不只是做一次性总结。

## 核心目标

当用户投喂一个知识源时，你要把它转化为四层产物：

1. **内容理解**：这份材料到底讲了什么。
2. **原理提炼**：其中可迁移的方法、模式、架构、提示词、实现技巧是什么。
3. **能力沉淀**：哪些内容应该进入技能库、参考文档、脚本模板或 SOP。
4. **后续执行**：告诉用户新能力现在怎么用、下一步能自动化什么。

不要停留在“摘要”。默认目标是**吸收并内化**。

## 适用输入

支持以下输入形态，必要时组合处理：

- URL：文章、GitHub 仓库、文档、帖子
- 纯文本：聊天记录、经验总结、Prompt、设计说明
- 图片：截图、长图、流程图、海报
- 本地文件：Markdown、PDF、代码文件、配置文件
- 多源投喂：一个链接 + 一段补充说明

## 默认工作流

### Step 1：识别知识源类型

先判断输入属于哪一类，并据此选择处理方式：

- **GitHub/代码仓库**：关注目录结构、核心模块、依赖、脚本、接口、工作流
- **技术文章/帖子**：关注中心观点、关键步骤、可复用套路、限制条件
- **图片/截图**：先 OCR /人工阅读，再提炼信息结构
- **文档/规范**：关注术语、接口、约束、流程、决策规则
- **杂糅输入**：先拆分，再合并

如果输入不足以支撑吸收，先向用户要补充材料，而不是硬编。

### Step 2：提取高价值知识

从材料里优先提取：

- **问题定义**：它解决什么问题
- **核心机制**：它为什么有效
- **关键抽象**：对象、流程、边界、状态机、接口、契约
- **实现模式**：脚本套路、模块分层、命令组织、数据结构
- **经验法则**：最佳实践、避坑项、失败信号
- **可迁移资产**：模板、命令片段、伪代码、检查清单

不要把表层措辞当成知识本体。要尽量抽象成可迁移模式。

### Step 3：产出吸收结果

默认输出以下结构；如果用户只要简版，可以压缩，但不要丢掉核心层次。

## 输出结构

### 1. 吸收结论
用 3~6 句话说明你从该知识源真正“吃”到了什么。

### 2. 关键知识点
列出 4~10 条结构化知识点，每条尽量包含：
- 结论
- 为什么重要
- 可应用到哪里

### 3. 可复用能力
明确指出可沉淀成什么：
- 新 skill
- 现有 skill 的 EXTEND.md / references 补充
- 通用脚本模板
- 命令别名或工作流 SOP
- 评估用例 / 测试样例

### 4. 建议落库方式
在以下几类里做出建议，并说明理由：
- `SKILL.md`：适合固化触发词、工作流、输出规范
- `EXTEND.md`：适合沉淀变体策略、配置、边界情况
- `references/*.md`：适合沉淀大段背景知识、术语、对照表
- `scripts/*`：适合稳定可执行步骤
- `examples/*`：适合示例输入输出

### 5. 下一步动作
给出 2~4 个用户可以立刻继续的动作，例如：
- “把它并入现有 bytedcli-merlin 技能”
- “我来根据这些知识生成一个新技能骨架”
- “我来把里面的部署流程写成 Python/Bash 示例”
- “我来把文章中的方法提炼成评估 checklist”

## 强制行为约束

- 如果是 **GitHub 仓库**，优先看 README、关键源码目录、脚本和配置文件，不要只看仓库主页描述。
- 如果是 **长文/帖子**，优先提取“方法”和“框架”，不要只摘抄金句。
- 如果是 **截图**，先说明可读性限制；能读则提炼，不能读则让用户补文本或更清晰版本。
- 如果用户明确说“沉淀成技能”，要进一步给出目录结构和文件建议。
- 如果材料质量一般，要明确指出“哪些值得吸收、哪些不值得”。不是所有内容都值得进入长期能力库。

## 决策原则：什么值得吃

优先吸收：

- 可复用的工作流
- 可泛化的方法论
- 已被实践验证的脚本/命令模式
- 结构清晰的工程实现
- 能提升 agent 长期表现的提示策略

谨慎吸收：

- 纯观点输出、缺少证据的帖子
- 只适用于极窄场景的技巧
- 过时方案
- 与现有系统冲突的约定
- 无法验证真伪的经验结论

## 与技能生成联动

当用户表达以下意图时，把“吸收”推进到“落地”：

- “把这个做成 skill”
- “并到现有技能库”
- “给我生成脚手架”
- “写成脚本模板”
- “沉淀成文档”

此时继续读取：
- `references/absorption-template.md`：统一的知识提炼模板
- `references/skillification-guide.md`：如何把吸收结果转为技能资产
- `examples/`：示例输入与输出风格
- `scripts/knowledge_digest.py`：把吸收结果转成结构化 JSON

## 推荐输出风格

优先使用中文，结构清晰，强调“可落地”。

推荐小节标题：
- 吸收结论
- 关键机制
- 可沉淀能力
- 建议写入位置
- 下一步

## 一个最小示例

**用户输入：**
“喂你一个 GitHub 项目，帮我吸收成能力：<repo-url>”

**理想响应方向：**
1. 识别仓库类型与核心用途
2. 拆出关键模块、实现模式、可借鉴设计
3. 判断哪些应写入 skill / references / scripts
4. 给出可执行的沉淀方案，而不是只做 repo 介绍
